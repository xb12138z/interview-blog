## AI项目：

**1.文档怎么切分？**

对于说明书、售后描述这类连续性较强的文档，我采用固定长度切分，chunk 大小大约 500 tokens，并设置 15% 左右的 overlap，主要目的是避免语义恰好落在 chunk 边界时造成信息断裂。
对于 FAQ、标准问答这类结构化知识，我采用按 QA 对切分，把单条问答作为最小知识单元。这样做的原因是售后问答场景本身就很适合以完整问答对作为检索单元，通常比单纯按长度切分匹配效果更稳定。
切分完成后，我会对每个 chunk 做内容去重，比如通过 MD5 判断是否已经入库，避免重复知识片段污染检索结果。

**2.embedding 用的什么？**

向量化这块我用的是 embedding-v4 模型。
选择它主要是因为它对中文语义检索场景支持比较好，适合售后问答这类知识匹配任务。实际流程是先对切分后的知识片段做 embedding，用户问题进来后也做同样的向量化，再在向量库中做相似度检索。

**3.相似度怎么算？**

因为 embedding 之后，文本会被表示成高维向量，余弦相似度更适合衡量两个向量在语义空间中的方向接近程度，也就是语义相似程度。
在检索阶段，我会把用户问题向量和知识库中的 chunk 向量做相似度比较，再按分数排序取 TopK 结果作为候选知识片段。

**4.向量库存在哪**

向量库这块我当前使用的是 ChromaDB，本地部署存储。（目前我的ChromaDb采取了一个userid对应一个collection的设计方案，适用于中小型规模的用户使用（比多用户一个库相比查询时不用where检索）。但当用户规模变大时我会考虑对用户id进行hash分组，分到不同的collection中，然后再通过where搜索对应的id。 （但面向多公司时，可用org公司名进行路由，路由到对应公司的数据库进行检索））  
目前这样做主要是为了方便开发和调试，后续如果面向更大规模数据量，可以替换成更适合生产环境的向量数据库，比如 Milvus 或 pgvector。

**5.召回几条**

当前我一般先召回 TopK，K 取 10 条。  
之所以不取得太多，是因为售后问答场景中相关知识通常比较集中，如果召回太多会把噪声一并带进提示词，反而影响最终生成效果。  
实际项目里这个 K 值我会作为可调参数，根据知识库规模和问答效果做调优。

**6.有没有 rerank**

cross-encoder
它的思路是：把“问题 + 候选文档”一起输入模型，模型直接判断这个候选和问题的匹配程度。  
优点：通常比 embedding 检索更准。  
缺点：更慢、不适合直接对全库跑、更适合在 TopK 结果上做精排  
我目前采用的就是阿里百炼（qwen3-rerank），“我会把问题和召回到的候选文本送进 rerank 模型，让它输出更精确的相关性排序分数。”

**8.你这个项目和普通“调用大模型 API 的聊天页面”相比，技术亮点在哪里？**

我觉得亮点主要有三点。第一，它不是单纯转发用户问题，而是支持多会话上下文管理。第二，它接入了 RAG，把说明书、FAQ、售后知识先做知识化处理，再辅助大模型回答。第三，它支持 MCP 风格的工具调用，让模型不只是“会聊天”，还可以借助工具完成更准确的回答。

**9.服务启动后做了哪些初始化工作？顺序为什么这样安排？**

启动后我会先初始化核心基础设施，包括 HTTP 服务、数据库连接池、RabbitMQ 连接和线程资源。然后我会把历史聊天记录从 MySQL 恢复到内存中的会话对象里，最后再启动消息消费线程和 HTTP 监听。这样安排的目的是先把依赖资源准备好，再把上下文预热好，最后再对外提供服务，避免服务刚启动时大量请求直接打到冷数据上。

**为什么要在启动时把历史聊天从 MySQL 读到内存？**

因为mysql是一种数据持久化的存储方式，但是如果每次问答时我都将数据存储到Mysql中，我在对话时传入历史聊天记录的时候会花费较长的时间在历史聊天记录的读取上，这样会导致我在封装提示词的时候变慢。用户获得聊天的回答速度变慢，用户体验不好。

**从用户发送一条消息到拿到 AI 回复，完整链路是怎样的？**

从后端服务器开始，首先我会读取到用户发送的一条消息，然后我对消息进行tcp、http解析，之后获得用户的消息的具体要求。解析完毕后会根据消息中的路由地址触发指定地址的路由，如果是发送的消息的话，我们会进入AIHELPER模块，模块中我们会从内存中找到对应用户存储的聊天记录，并且将用户的问题与我们之前建立好的chroma数据库中存储的数据进行匹配（向量匹配和rerank），之后我们会根据用户使用的大模型是否支持MCP来决定是否将工具的要求写入到提示词模板中。最后我们封装好提示词发送给大模型获得回答。如果调用了MCP的话，我们还需要循环监听大模型是否完成了工具的调用，完成之后再输出大模型的调用结果。之后我们会将会话打入到rabbitMQ中进行数据库的写入，同时我们会将数据分装好HTTP请求，发送给用户。

**新建会话和已有会话继续聊天，这两条链路有什么区别？**

已有的会话聊天时，我们会直接进行内存的读取和用户的对话。新建对话时我们会给对话先分配一个sessionID，保证用户对话可以追踪。同时我们还会再首次对话新建对应的数据库中的行，来记录用户的回答消息。

**1.你能用一句话描述 ChatServer、GateServer、StatusServer 三者在这个系统里的职责和调用链吗？**

GateServer主要用于客户端与服务器的初次通信，比如说用户的注册登录等，主要负责用户的管理。在用户成功登录后将用户分配给statusServer。[负责客户端TCP/长连接接入、协议解析、鉴权/登录、会话管理(心跳检测)和把用户路由到后端业务节点。]
StatusServer主要负责与用户状态的交互，比如说添加好友等功能，当用户需要进行聊天时用户会被分配给最少承载量的chatServer[负责用户基本信息与关系（在线状态、好友、群、权限）管理，并对外提供路由/分片决策]
ChatServer主要负责维护用户之间的沟通网络渠道进行用户消息的转发[负责具体消息通道（点对点、群组）传输、消息可靠投递、离线缓存与转发。]

**2.为什么这里用 gRPC (ChatServiceImpl) 同时又用原生 boost::asio TCP 自定义消息协议？这两种方式你如何决定适用场景？**

gRPC是谷歌开发的RPC协议的标准格式，一般用于协议的解析当客户端要传输明确的函数调用指令时我会用gRPC进行传输，以便更好地执行回调函数。TCP自定义消息协议我主要用于服务器与mysql和redis的交流中和客户聊天传输中还有客户端状态返回中，这些不需要明确调用服务器函数的场景中

错误点：
客户端与 MySQL/Redis 不会直接用自定义 TCP：MySQL/Redis 都有各自协议/客户端库，你代码里是用 RedisMgr/MysqlDao 直接 API 调用，不是 CSession 的 socket。
gRPC 不只是“执行回调函数”，而是“在进程间做严谨的IDL定义、序列化/负载、双向流、服务发现和易扩展”。
ChatServiceImpl 其实用于跨进程/跨服务的后端通信（比如状态服/网关/聊天服之间的 RPC），而 boost::asio 主要用于客户端 TCP 长连接的数据通道和自定义消息包。

**标准答案：**
gRPC适用于后端服务间调用（如网关与状态服、状态服与消息服之间的远程接口），因为它有IDL、自动序列化、流控、超时和简单可监控。boost::asio 自定义TCP用于客户端长连接消息通道（聊天协议、心跳、自定义二进制包），因为它更轻量、延迟更低、可控更强。

**3.AsioIOServicePool 你是怎样设计线程/IO分配的？如何避免 io_context 竞争与线程切换开销？**

AsioIOServicePool是线程池，在池中我有很多线程通信接口资源。我采取了一个主ioc监听，然后对其分配池中资源在分配时我会判断池中是否有可用资源，如果没有的话我会让当前任务在队列中进行等待池中资源空出，在池中资源排序部分我会用队列格式尽量少让一个线程资源承担大量的任务实现负载均衡。

**标准答案：**
在你这个项目里，AsioIOServicePool 是一个固定大小的 io_context 池，每个 io_context 绑定一个线程执行 run()。核心逻辑：

1.构造时根据 size 创建 size 个 boost::asio::io_context 和对应 executor_work_guard，确保 run() 不会因为没有任务而退出。
2.每个 io_context 启动一个独立线程 thread[i]，并在该线程内执行 _ioServices[i].run()。
3.GetIOService() 使用 round-robin 轮询 _nextIOService，把新会话/异步操作分配到下一个 io_context，实现简单负载分摊。
4.关闭时调用 stop() 并 join() 所有线程。

**7.CServer::_sessions 是 unordered_map + mutex，你会考虑读写锁/细粒度锁或者 concurrent_hash_map 吗？为什么？**

错误：

你这句“CServer::_sessions 只是存入，不需要只读”不准确
_sessions 在现实里会有：新连接插入、断开删除、查找会话（例如推送、清理）

这些操作并发出现，且读操作通常比写多得多（例如按 session_id 查找推送、心跳处理）。

具体为什么要考虑读写锁/细粒度/concurrent map
mutex 每次全表排它锁：高并发下容易成为瓶颈。
读写锁（shared_mutex）对“多读少写”非常适合。
细粒度锁（分段哈希）可进一步降低冲突。
concurrent_hash_map（TBB、folly）能直接支持高并发插入/查询，性能大幅提升。

业务访问路径少写多读：会话查找、发送等是读，断开/新增写；读写锁能显著提高并发吞吐。
最佳做法：把 _sessions 分片为 N 个 bucket，每个 bucket 一把锁，插入/删除/查询只锁对应 bucket（细粒度）。

**8.LogicSystem::PostMsgToQue 应该跨线程提交业务消息，你能讲讲你这部分的线程模型？如何确保顺序一致与消息回调线程安全？**

你现在 LogicSystem 设计是：

CSession 的 AsyncReadBody 在 IO 线程里收到一条消息后，PostMsgToQue() 把 LogicNode 推到业务队列
LogicSystem 启动一个 std::thread 运行 DealMsg()，这是单线程消费业务消息
所以整体模型是“一个生产者线程池（多个IO线程）+一个消费者线程（业务处理单线程）”

线程模型:

生产者：IO线程
每个连接的 async_read 回调运行在某个 io_context 线程回调只做数据解析、构造 LogicNode，然后 PostMsgToQue

线程安全队列
PostMsgToQue 用 _mutex + _msg_que + _consume 条件变量 , 一条条消息入队，消费线程被唤醒

消费者：业务线程
DealMsg 中循环取队头并执行回调 , 这个线程单一，业务逻辑无竞态（只要内部不越界访问共享全局）

如何保证顺序一致:
因为 PostMsgToQue 只按接收顺序 push，DealMsg 按队列 front/pop 处理，消息顺序在单节点内保持 FIFO。
如果你要保证“同一用户顺序”，还要确保同一 uid 的消息从同一个 CSession 单连接发来，这里已经满足。
如果你跨多个连接与多个 uid 并发，顺序在每个连接内是保证的，但不同用户间无全局顺序要求。

如何保证线程安全:
生产者与消费者之间通过 mutex+condvar，这是线程安全。
业务回调 (_fun_callbacks[...]) 运行在单业务线程，避免在此处加锁，但它会调用 UserMgr/RedisMgr等共享资源，这些需要各自线程安全（例如 RedisMgr 内部线程安全）。
回调中若调用 session->Send，CSession::Send 内有 _send_lock 保护写队列，安全。

你当前模型的风险:
当前只有一个 LogicSystem 业务线程：会成为瓶颈，建议改成多消费者 worker thread pool 或按 uid 分槽路由。
不同消息可能需要不同优先级（心跳/控制 vs 聊天），单队列无优先级。
PostMsgToQue 发起频率极高时 _mutex 竞争会成为 bottleneck；可改成多队列/lock-free ring buffer。
长时间业务处理（数据库慢）会“阻塞业务队列”，建议加异步 DB/GRPC回调拆分。

**11.UserMgr::RmvUserSession 这类跨容器状态清理，如何保证同一用户多连接、断线重连、并发登出时一致性？**

uid -> single session 映射（一个 uid 只保存一个 session）[session作用见19]
SetUserSession 直接覆盖，没有多连接管理
RmvUserSession 有检查逻辑被注释了，直接删除可能"删错"
锁保护不足以处理多连接场景

当前问题：

多连接覆盖：同一用户第二个连接 SetUserSession 会覆盖第一个；第一个断开清理时会把第二个也清了。
断线重连：旧session vs 新session，无法同时清理两者。
并发登出：多个 session 同时调 RmvUserSession，可能出现 race condition。
缺少幂等性：重复登出同个 uid，第二次操作会出错或无效。

最优解决方案（3 层）：
第一层：数据结构改进
```cpp
// 一个 uid 对应多个 session（按优先级/最新）
std::unordered_map<int, std::vector<std::shared_ptr<CSession>>> _uid_to_sessions;
// 或者按 session_id 做反向映射
std::unordered_map<std::string, int> _session_id_to_uid;
```

第二层：清理逻辑
```cpp
void RmvUserSession(int uid, std::string session_id) {
    std::lock_guard<std::mutex> lock(_session_mtx);
    auto iter = _uid_to_sessions.find(uid);
    if (iter == _uid_to_sessions.end()) return;
    
    // 删除该 session_id 对应的连接
    auto& sessions = iter->second;
    sessions.erase(
        remove_if(sessions.begin(), sessions.end(), 
            [&](const auto& s) { return s->GetSessionId() == session_id; }),
        sessions.end()
    );
    
    // 如果该uid没有会话了，清理 Redis 中的状态
    if (sessions.empty()) {
        _uid_to_sessions.erase(uid);
        RedisMgr::GetInstance()->Del(USERIPPREFIX + std::to_string(uid));
    }
}
```
第三层：多连接与重连策略
```cpp
void SetUserSession(int uid, std::shared_ptr<CSession> session) {
    std::lock_guard<std::mutex> lock(_session_mtx);
    
    // 保存新会话
    _uid_to_sessions[uid].push_back(session);
    
    // 可选：如果超过最大连接数（如3个），删除最旧的

        std::to_string(GetTotalSessionCount()));
}
```

分布式与并发控制：
使用分布式锁防止并发冲突：

**14.gRPC 的 server->Wait() 和 io_context.run() 是两个线程，在退出流程中有 race condition 吗？你会怎么做优雅关闭、停机阶段的“拒绝新请求+等待完成”？**

grpc主要用于服务器内部通信，用于处理chatserver与statuserver的信息交流，用于statuserver分配后的注册。之后io_context线程只用于与客户端交流。处于停机阶段是我会让chatserver给statuserver发送停机信号，statuserver发送别的chatserver的ip端口让客户端可以切换到别的chatserver中

错误点：有Race。 Race Condition 分析：

有race：grpc::Server::Wait() 和 io_context.run() 在不同线程并发；退出时若不同步，一个线程还在处理请求，另一个已清理资源，可能导致崩溃或数据竞态。

💡 最优优雅关闭流程（“拒绝新请求+等待完成”）
信号捕获：收到SIGINT/SIGTERM，设置“关闭标志”。
拒绝新请求：
gRPC：调用 server->Shutdown()，拒绝新RPC。
TCP：停止 acceptor（acceptor.cancel()），拒绝新连接。
等待完成：
gRPC：server->Wait() 等待所有RPC完成。
TCP：io_context.stop() 并等待 run() 退出；用 std::thread::join() 确保线程结束。
清理资源：断开所有会话、清理队列、释放锁。
分布式通知：如你所说，通知StatusServer更新路由表，让客户端重连到其他ChatServer。
同步退出：用 std::promise 确保两个线程都完成后再退出主进程。

标准回答：

会有race condition，我用信号同步：先Shutdown gRPC和stop io_context拒绝新请求，然后Wait/Join等待完成，最后通知StatusServer切换客户端，确保无遗漏。

**18.如果要上线灰度、A/B 测试、按业务线限流（例如带宽 vs 消息频率），你会在哪层做限流：网关、会话、logic、redis计数那层？**

网关（GateServer）：全局入口，适合灰度/A/B（按用户ID/百分比路由）、全局限流（QPS/带宽）。
会话（CSession）：连接级，适合消息频率限流（token bucket per session）。
Logic：业务级，适合复杂规则（e.g., 按业务线限流）。
Redis计数：分布式，适合跨实例限流（e.g., 用户全局频率）。

最优策略：
灰度/A/B测试：网关层，按 uid % 100 或 feature flag 路由到新版本。
按业务线限流：logic层（业务逻辑限流） + Redis（分布式计数），e.g., 消息频率用滑动窗口。
带宽限流：会话层（连接带宽控制） + 网关（全局带宽池）。

标准答案：
“灰度/A/B在网关路由，限流分层：全局QPS网关，会话频率logic+Redis，带宽会话+网关，确保灵活可控。”



**19.session是干什么用的？**

CSession 代表一个"客户端连接会话"，核心作用：

持有TCP连接：每个客户端连上来开一个 socket，一个 CSession 对应一个 socket。
异步收发：通过 AsyncReadHead/AsyncReadBody 接收客户端消息，通过 Send 向客户端发送消息。
会话标识：有唯一 session_id (UUID)，用于追踪和清理。
用户绑定：连接后调 SetUserId(uid)，把这个连接和某用户关联。
在你的聊天系统里：

客户端A连接 → 创建 CSession_A → 绑定 uid=101 → UserMgr 记录 uid:101 -> CSession_A
**有消息来**→ CSession_A 读到消息 → POST 到业务队列 → LogicSystem 处理
**发消息给A** → UserMgr.GetSession(101) 拿到 CSession_A → session->Send(msg) 发回去
**断连** → CSession_A.Close() → CServer.ClearSession() → UserMgr.RmvUserSession(101) 清理映射
所以 UserMgr 中维护的 uid -> session 映射，就是"给定一个用户id，能否快速找到他当前的连接，把消息发给他"。

**多连接问题就是：同一用户如果有多个设备/窗口登录，需要管理多个 session**
